# Connected Components Workbench (CCW) workflow & idiom

CCW is Rockwell's IDE for the Micro800 family. It's a closed GUI tool with proprietary binary
formats and several non-obvious behaviors. This is the operational knowledge for getting code
from a `.stf`/`.st` source into a running, talking controller.

## Source vs binary

| File | What it is | Editable? |
|---|---|---|
| `*.stf` | CCW program source (Structured Text or Ladder, proprietary container) | via CCW; the readable ST text is recoverable but the file is not a plain `.st` |
| `*.st`  | hand-maintained plain-ST **shadow** of a ladder POU (translator-readable mirror) | yes, by hand — keep in sync with the `.stf` until the QLD parser ships |
| `GlobalVariable.rtc` | compiled Controller Variables table (binary) | **no** — regenerated by CCW on build |
| `*.rtc` / `*.out` | compiled artifacts downloaded to the controller | no |

You author the **source-of-truth artifacts** (`.stf`, the GlobalVariable spec, `MbSrvConf.xml`)
and CCW regenerates the `.rtc` binaries on rebuild. A documentation pattern that works: keep a
human-readable `GlobalVariable.md` table next to the `.stf` describing exactly what the
Controller Variables table must contain, and apply it by hand in CCW.

### Verify the rebuilt binary

After applying a variable-table change and rebuilding, sanity-check the binary with `strings`:

```bash
strings GlobalVariable.rtc | grep -iE 'MSG_MODBUS|MSGMODBUSPARA|READDATA|read_data'
# expect:        MSG_MODBUS  MSG_MODBUS_LOCAL  MSG_MODBUS_TARGET  read_data
# must NOT see:  MSG_MODBUS2  MSGMODBUSPARA_*  READDATA   (the TCP variant / the dead scalar)
```

## Global Variables — no `VAR` blocks in the POU

Micro800 POUs **do not declare local `VAR … END_VAR`** the way you might expect. Every variable
a program references must exist in the **Controller → Global Variables** table first, with the
right name, data type, initial value, and retain flag. Writing `VAR` blocks in the source
produces cryptic errors. Treat the Global Variables table as the single declaration point.

## Build → download → go online

1. **Build** (Project menu). Fix errors top-down.
2. **Connect** to the controller and **Download** the program.
3. **Go online / Monitor** to watch variables live in the watch table — this is your primary
   verification surface (Micro820 has no real-time breakpoints; instrument with counters/flags).
4. **Serial framing is set here, not in ST:** Project → Micro820 → Embedded Serial Port →
   Properties: driver = Modbus RTU Master, baud / data / parity / stop / media (RS-485) /
   response timeout. This must match the GS10's `P09.xx` exactly (see `gs10-register-map.md`).

## Gotchas (each one has cost a debug session)

- **`ErrorID 255` = config never downloaded.** If a MSG instance never completes (`.Q` and
  `.Error` both stay FALSE), the serial-port configuration didn't actually reach the
  controller. Re-download. This is a CCW sync problem, **not** a wiring fault.
- **Channel name drifts across firmware revs.** The embedded RS-485 port is `Channel 0` on the
  bench-proven MIRA_PLC build (and in `device-profiles/micro820.yaml`); the monorepo v4.x
  lineage used `Channel 2`. Wrong channel → **zero TX activity.** If you see no traffic on the
  wire, try the other channel value.
- **Cascade errors from one undeclared identifier.** A single variable missing from the Global
  Variables table makes the parser emit 5–10 downstream syntax errors (missing `;`, unexpected
  `END_IF`, …). Fix the **first** undeclared-identifier error, rebuild, and the rest usually
  vanish. Don't chase the downstream noise.
- **Dead-scalar landmine.** Leaving an old scalar global (e.g. `READDATA`) declared alongside
  the array you actually use (`read_data`) invites a future case-typo to silently bind to the
  wrong storage. Delete obsolete globals when you rename them (the F4 lesson).
- **Modbus/TCP server returns exception 1 until the map is deployed.** The Micro820's
  Modbus/TCP side answers `ILLEGAL_FUNCTION` on every read until `MbSrvConf.xml` is built and
  downloaded. This is separate from the RS-485 RTU client side.

## Modbus/TCP server map (`MbSrvConf.xml`)

The Micro820 also exposes a Modbus/TCP **server** so a monitor (e.g. `live_monitor.py`) can read
state without touching CCW. Map a small set of internal variables (motor speed, error code, VFD
frequency, `conv_state`, `vfd_cmd_word`, `poll_step`, the `commissioning_mode` coil) to fixed
offsets. Keep that mapping in `MbSrvConf.xml`; it's the contract the monitoring tools read against.

## Remote programming

The PLC laptop runs CCW (closed GUI, no CLI build) and is reached over Tailscale. Two notes
that have bitten before:

- **Tailscale route hijacking:** if a peer advertises the `192.168.1.0/24` subnet and the laptop
  has `--accept-routes` on, PLC traffic tunnels through Tailscale instead of the direct cable —
  TCP/502 and TCP/44818 go unreachable. Fix: `tailscale set --accept-routes=false` on the laptop.
- **Liveness check without a program loaded:** an EtherNet/IP CIP **List Identity** query (UDP
  44818) confirms the controller is alive even when no Modbus map is deployed — useful before a
  download attempt. (That probe is the read-only `fieldbus-discovery` job; this skill just relies
  on it.)

## Source-of-truth ordering (when facts disagree)

1. The **codified device profiles** — `device-profiles/gs10.yaml`, `device-profiles/micro820.yaml`
   (the KB with gotchas baked in).
2. The **actual `.stf`/`.st` source you opened** with your own eyes.
3. Any summary, older doc, or memory of it.

When (1) or (2) contradicts (3), the profile and the file win. And on the MSG_MODBUS RTU/TCP
axis specifically, distrust older MIRA docs — that fact has been inverted repeatedly.
