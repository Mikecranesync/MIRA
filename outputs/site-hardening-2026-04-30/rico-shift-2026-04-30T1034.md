## What broke

1. **Off-topic scenario picked** — undergrad research question, not a fault report. Corpus filter is too loose.
2. **Rico's followups are generic fillers** (`correct`, `yep`, `yep`) — `_generate_follow_up()` in `rico.py:378` keyword-matches on MIRA's reply and falls through to a 5-item generic-acknowledgement list when no keyword hits. Nothing in MIRA's question "What kind of equipment are you trying to fix?" matched the Rico keywords (voltage / temperature / load / reset / error / history), so Rico shrugs.
3. **MIRA loops asking for the model number** because Rico never provides it. Conversation never reaches diagnosis or work-order creation.
4. **`fsm_advanced` returns True** because final reply length > 50 and no `?` in first 50 chars — but that's just heuristic on shape, not actual progress. Rico marks the run "successful" while the real outcome was a dead loop.
5. **Telegram shift report uses `parse_mode=Markdown`** with no fallback (rico.py:`_tg_send`). MIRA's reply contains unbalanced `*` (the truncated `*Lo`); if Rico ever included MIRA's raw reply in the Telegram body, it would 400 the same way the agent orchestrator did — fixed for the orchestrator in PR #886, NOT fixed for Rico.
