01

System Overview

Charlie HUD is a browser-based heads-up display for floor-level maintenance technicians. It provides live webcam monitoring, guided maintenance procedures, step-by-step checklists, and an optional AI assistant panel powered by MIRA.

HUD Server

A lightweight Node.js process that serves the HUD interface on localhost:3000. Handles webcam relay, procedure loading, and proxying AI queries to the MIRA backend.

MIRA AI (Optional)

A separate Python backend running an LLM via Ollama. Answers maintenance questions in plain language using equipment manuals, fault codes, and procedure knowledge. HUD works fully without it — AI panel shows OFFLINE if not running.

02

Requirements

Required

  • macOS 12 or later (Monterey, Ventura, Sonoma, Sequoia)
  • Node.js via Homebrew — installed at /opt/homebrew/bin/node ALREADY INSTALLED
  • A modern web browser — Safari, Chrome, Firefox, or Arc

Optional (for AI answers)

  • MIRA AI backend — Python FastAPI service on localhost:8000
  • Ollama — local LLM inference engine on localhost:11434

The HUD runs perfectly without these. The AI panel will show OFFLINE until the backend is started.

03

First-Time Install

Run the installer once — this sets everything up and puts the app icon on your Desktop.

  1. Open Terminal (press ⌘Space, type "Terminal", press Enter)
  2. Navigate to the project: 
    cd ~/Documents/GitHub/MIRA/mira-hud
  3. Run the installer: 
    bash install.sh
  4. The script will:
    • Verify Node.js is installed
    • Make the launcher executable
    • Install npm dependencies if missing
    • Copy CharlieHUD.app to your Desktop
    • Optionally install to /Applications
    • Offer to launch immediately

Gatekeeper Warning (first launch only)

macOS may say "CharlieHUD cannot be opened because the developer cannot be verified."

This is expected for apps not signed with an Apple Developer ID. To allow it:

  1. Go to System Settings → Privacy & Security
  2. Scroll down to the Security section
  3. Click "Open Anyway" next to the CharlieHUD message
  4. You only need to do this once

Alternatively: right-click the app → Open → click Open in the dialog.

04

Daily Launch

After the one-time install, launching is simple:

  1. Double-click CharlieHUD on your Desktop (or in /Applications)
  2. A macOS notification fires: "Server started on port 3000"
  3. Your default browser opens automatically to http://localhost:3000
  4. The HUD boot sequence runs (green text animating in terminal-style)
  5. The interface loads — webcam activates, clock starts ticking

What happens under the hood

  • Any previous HUD server instance is automatically killed (no "port in use" errors)
  • The Node.js server starts in the background — no terminal window appears
  • Server logs are written to /tmp/charlie-hud.log
  • The app icon appears in the Dock while the process is running

Server stays running

Closing the browser tab does not stop the server. The server continues running until you explicitly stop it (see Section 8). You can reopen the HUD anytime by going to http://localhost:3000 in your browser.

05

Using the HUD

Webcam Panel

The left panel shows a live webcam feed from your built-in or USB camera. This is intended for visual documentation — point your camera at the equipment you're working on.

  • Your browser will ask for camera permission the first time — click Allow
  • If no camera is detected, the panel shows NO SIGNAL
  • The feed is local-only — nothing is transmitted or recorded

Procedure Selector

The top dropdown lets you select a maintenance procedure. Each procedure is a numbered checklist of steps loaded from the server.

  • Select a procedure from the dropdown to load it
  • The step counter resets to Step 1
  • Steps are shown one at a time in the main panel

Step Navigation

Navigate through procedure steps using the PREV and NEXT buttons:

  • NEXT — advance to the next step
  • PREV — go back to the previous step
  • The progress bar at the top fills as you advance through the procedure
  • Current step and total count shown as STEP X / Y

Progress Bar

The green progress bar at the top of the procedure panel reflects your position in the current checklist. It fills incrementally with each NEXT press and empties on PREV. Selecting a new procedure resets it to 0%.

06

MIRA AI Panel

The right-side panel is the MIRA AI query interface. Type any maintenance question and MIRA will respond using equipment manuals, fault codes, and procedure knowledge.

Connection States

  • ONLINE — MIRA backend is running and responding
  • CONNECTING — checking connection to backend
  • OFFLINE — backend not running. HUD still works; AI answers unavailable

Response Tags

AI responses are tagged to indicate confidence and source:

  • VERIFIED — answer drawn from known equipment documentation
  • INFERRED — best-match answer, may need field verification
  • UNCERTAIN — low confidence; escalate to senior tech or OEM support
  • PROCEDURE — response includes a step-by-step action sequence

Example Queries

  • "VFD shows fault code F0001 — what does it mean?"
  • "Motor won't start after E-stop was released — what do I check?"
  • "What are the GS10 Modbus RTU parameters for 9600 baud?"
  • "How do I reset the fault latch on the Micro820?"
07

Starting MIRA AI Backend

The MIRA AI backend is optional and runs separately from the HUD server. Start it when you need AI-assisted answers.

Prerequisites

  • Ollama installed and running: ollama serve
  • A model pulled, e.g.: ollama pull llama3.2
  • Python dependencies installed in the MIRA project

Start the Backend

cd ~/Documents/GitHub/MIRA
python -m uvicorn mira.main:app --port 8000 --reload

Once running, the MIRA panel in the HUD will switch from OFFLINE to ONLINE within a few seconds (the HUD polls automatically).

Using the SCADA Stack (Docker)

If you need the full SCADA stack (Modbus sim, PLC API, diagnosis service):

cd ~/factorylm
docker compose up -d --build

Health check: curl localhost:8001/api/health

08

Stopping the HUD

The HUD server runs as a background process. Here are your options to stop it:

Option A — Relaunch (auto-kills prior instance)

Double-clicking CharlieHUD again will automatically kill the running instance before starting a fresh one. No manual steps needed.

Option B — Terminal kill command

pkill -f "node server.js"

This stops the server immediately. The browser tab will show a connection error, which you can close.

Option C — Kill by port

# Find what's on port 3000
lsof -ti :3000

# Kill it
lsof -ti :3000 | xargs kill

Checking server logs

cat /tmp/charlie-hud.log

Logs are overwritten each time the server starts. Useful for diagnosing startup errors.

Troubleshooting

Symptom Likely Cause Fix
Browser shows "This site can't be reached" Server not started or crashed on startup Check cat /tmp/charlie-hud.log for errors. Try relaunching the app.
"Port 3000 already in use" (in log) Stale server process from a previous session Run pkill -f "node server.js" then relaunch. Or just double-click the app — it auto-kills.
Camera shows black / "NO SIGNAL" Camera permission denied or no camera connected Go to System Settings → Privacy & Security → Camera and enable access for your browser.
"Developer cannot be verified" on first launch App not signed with Apple Developer ID (expected) System Settings → Privacy & Security → Open Anyway, or right-click → Open.
Node.js not found error in install.sh Node.js not installed at /opt/homebrew/bin/node Run brew install node in Terminal. If brew is missing, install Homebrew first.
MIRA AI panel shows OFFLINE permanently MIRA backend not running Start the backend (see Section 7). HUD works fine without it.
HUD boots but shows no procedures Procedure files missing or server startup error Check /tmp/charlie-hud.log. Verify mira-hud/ directory is intact with server.js and hud.html.
Double-clicking app does nothing Launcher not executable (permissions issue) Run bash install.sh again from Terminal — it re-applies chmod +x.

Getting more debug info

# Check if server is running
ps aux | grep "node server.js"

# Check what's listening on port 3000
lsof -i :3000

# Full server log
cat /tmp/charlie-hud.log

# Test server responds
curl -s http://localhost:3000 | head -5
