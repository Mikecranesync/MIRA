1 The Big Picture

You're going to push a green button. A motor will spin. Whether the motor spins forward or reverse depends on where a 3-position selector switch is pointing when you press the button. Flipping the selector while the motor is running will stop it тАФ you have to press the button again to start it back up in the new direction. Pressing the e-stop kills it immediately. Releasing the e-stop alone does not restart the motor; you have to press the green button again.

You're going to build that behavior once, inside a sealed box of logic called a User-Defined Function Block (UDFB). The sealed box has labeled terminals on its outside: one for the green button, one for each direction-switch position, one for the e-stop status, one for "enable", and on the output side one terminal that drops out the VFD command word and one that lights the run lamp. Inside the box, the logic is the same every time. You wire one box up to one motor. If you ever need a second motor, you wire a second copy of the same box up to a second pushbutton + selector + lamp. Fixing a bug inside the box fixes both motors at once.

2 The Real-World Problem

The bench conveyor has one motor today: a 2-HP 3-phase induction motor wired through the GS10 VFD. The control panel has a green illuminated pushbutton, a 3-position selector switch (FWD / OFF / REV), and a mushroom e-stop. Today, the motor-control logic lives inline inside Prog2 ST as a 5-state machine (IDLE тЖТ STARTING тЖТ RUNNING тЖТ STOPPING тЖТ FAULT). It works. But it's a wall of CASE statements that all reference the same global tags, and the moment a second motor needs the same treatment, you're copy-pasting the entire state machine and search-and-replacing tag names.

This guide does two things at once: (1) teaches the canonical seal-in motor-starter pattern as a UDFB, and (2) refactors the deployed Prog2 ST so the new UDFB is the sole authority on vfd_cmd_word тАФ no more split-brain between the state machine and a new block fighting over the same global.

4 Why This Becomes a Problem

You wired five rungs. They work. Now imagine the conveyor gets a second motor тАФ say a discharge cross-belt тАФ that needs the same exact behavior: its own green PB, its own selector, its own e-stop (or sharing the main e-stop), its own run lamp, writing to its own VFD's command word over Modbus. To replicate the five rungs:

  • Five rungs become ten rungs.
  • Every tag in those new five rungs gets a "_2" suffix or its own new name.
  • You find a bug in the seal-in logic (say, you realize the dropping-direction rung doesn't account for the "both selector contacts closed = wiring fault" case). You fix it on motor 1's rungs. Two weeks later, motor 2 has the same fault and you spend an hour wondering why it doesn't behave like motor 1.
  • The five-rung pattern is now ten rungs; if a third motor appears, fifteen. Each is a copy of the same logic with slightly different tag names. The cognitive load to verify "yes, all three are doing the same thing" is no longer linear.
The pain made concrete This isn't hypothetical. The deployed Prog2 ST today has a 5-state machine that does roughly the same job, and adding a second conveyor motor would mean copy-pasting the whole CASE conv_state OF block with renamed locals. The state-machine approach is more powerful than the flat rungs above (timers, fault states, recovery transitions) but it suffers from the same scaling problem when motor-count grows.

5 The Better Pattern тАФ User-Defined Function Block

A UDFB is a sealed box of logic with labeled input and output terminals. You define it once тАФ name it, declare its inputs, declare its outputs, declare its private locals, write the body. From that point on, every place in the program that needs that behavior just drops the block in and wires the terminals. Inside, the logic runs identically every time. The terminals are the contract: as long as the caller wires the inputs to something sensible, the block does what it promises.

A UDFB is a sealed starter cabinet. You wire field devices to the terminals on the front тАФ start PB, direction switch, e-stop, run lamp, VFD command. Inside is the latching logic, identical in every cabinet. Build one cabinet, install two. Fix a bug in the cabinet design, every cabinet on the floor is now fixed.

The block has instances. Each instance is an independent copy of the internal storage (in our case: RunLatch, DirLatch, PrevStartPB). One motor's instance latches separately from the other's, even though both run the same block code. That's the leverage of the pattern.

Have CCW open with the MIRA_v5 project loaded. We're going to create one UDFB (motor_starter), call it once from a new ladder rung in Prog3 (or a fresh Prog4), and remove the conflicting vfd_cmd_word := assignments from Prog2 ST.

Step 3 тАФ Paste the ST body

Click into the ST editor and paste this verbatim. The body is intentionally short тАФ about 30 lines.

(* === motor_starter тАФ green PB seal-in with direction-lock + e-stop drop === *)

(* Rising-edge on start pushbutton тАФ prevents stuck-PB auto-restart *)
StartPBRising := StartPB AND NOT PrevStartPB;
PrevStartPB := StartPB;

(* Did the operator flip the selector to a different position while running? *)
DirChangedWhileRunning := RunLatch AND (
    (DirLatch = 1 AND NOT DirFwdSw) OR
    (DirLatch = 2 AND NOT DirRevSw)
);

(* Anything that should drop the seal-in latch *)
DropConditions := (NOT Enable)
               OR (NOT EStopOK)
               OR (NOT DirFwdSw AND NOT DirRevSw)   (* selector OFF = stop *)
               OR (DirFwdSw AND DirRevSw)            (* both contacts = wiring fault *)
               OR DirChangedWhileRunning;

(* Seal-in state machine *)
IF DropConditions THEN
    RunLatch := FALSE;
    DirLatch := 0;
ELSIF StartPBRising AND NOT RunLatch THEN
    IF DirFwdSw THEN
        RunLatch := TRUE;
        DirLatch := 1;
    ELSIF DirRevSw THEN
        RunLatch := TRUE;
        DirLatch := 2;
    END_IF;
END_IF;

(* Output assignment *)
IF NOT RunLatch THEN
    VFDCmdWord := 1;             (* STOP *)
ELSIF DirLatch = 1 THEN
    VFDCmdWord := 18;            (* FWD + RUN тАФ bench-verify, see cheatsheet *)
ELSIF DirLatch = 2 THEN
    VFDCmdWord := 20;            (* REV + RUN тАФ bench-verify, see cheatsheet *)
ELSE
    VFDCmdWord := 1;             (* defensive fallback *)
END_IF;
RunLamp := RunLatch;

Step 5 тАФ Add the calling rung in a new LD program

You can call the UDFB from Prog3 (where the Modbus rungs live), or тАФ cleaner тАФ add a Prog4 specifically for motor control logic. Either works. The walkthrough below assumes Prog4 for clarity.

  1. Project Organizer тЖТ Programs тЖТ right-click тЖТ Add тЖТ New LD: Ladder Diagram тЖТ name it Prog4-MotorControl.
  2. Set execution order: Prog2 first, Prog3-Modbus second, Prog4-MotorControl third. Properties тЖТ Order.
  3. Open Prog4. Drag a contact onto rung 1, type enable_motor_ctrl (a global BOOL you'll create; default TRUE).
  4. From Toolbox тЖТ User-Defined Function Blocks тЖТ drag motor_starter onto the rung. Instance name dialog: enter motor_starter_inst.
  5. Wire each pin by clicking the pin and typing the tag:
    • Enable тЖР (wired from the contact)
    • StartPB тЖР _IO_EM_DI_04
    • DirFwdSw тЖР dir_fwd
    • DirRevSw тЖР dir_rev
    • EStopOK тЖР NOT e_stop_active AND NOT estop_wiring_fault (you may need an inline expression rung; or pre-compute as e_stop_ok global in Prog2 ST)
    • VFDCmdWord тЖТ vfd_cmd_word
    • RunLamp тЖТ _IO_EM_DO_00
"EStopOK" can't be a NOT-NOT expression on a ladder pin CCW won't let you type NOT e_stop_active AND NOT estop_wiring_fault directly into the pin field тАФ it expects a single variable. Easiest fix: add one line to Prog2 ST: e_stop_ok := NOT e_stop_active AND NOT estop_wiring_fault; and declare e_stop_ok : BOOL; as a global. Then wire EStopOK тЖР e_stop_ok.

Step 6 тАФ Remove the conflicting vfd_cmd_word := in Prog2 ST

Per the design decision Mike locked when approving this guide: the UDFB drives vfd_cmd_word directly. The old state-machine assignments must be removed (or commented out) so both don't write to the same global within one scan.

  1. Open Prog2 in CCW.
  2. Find the CASE conv_state OF block.
  3. Comment out (with (* ... *)) every line that assigns to vfd_cmd_word. There should be three such lines: vfd_cmd_word := 1; (in STOP/IDLE/FAULT cases), vfd_cmd_word := 18; (in STARTING/RUNNING + dir_fwd), vfd_cmd_word := 20; (in STARTING/RUNNING + dir_rev).
  4. Leave the rest of the state machine alive. conv_state transitions, start_timer, motor_running, item_count тАФ all keep working. Only the vfd_cmd_word := lines move to the UDFB.
  5. Save Prog2.stf, build the project (F8), download.
If you skip Step 6 and both Prog2 ST and Prog4 LD assign to vfd_cmd_word in the same scan, the last writer wins тАФ which depends on program execution order. With order Prog2 тЖТ Prog3 тЖТ Prog4, the UDFB's value would win for one scan, then Prog2 ST overwrites it on the next, and the motor behaves randomly. Comment the old lines now and never wonder later.

8 Code Walkthrough тАФ chunk by chunk

Chunk 1 тАФ Edge detection on the start pushbutton

StartPBRising := StartPB AND NOT PrevStartPB;
PrevStartPB := StartPB;

The first line computes "was the button low on the previous scan AND is it high this scan?" тАФ that's the definition of a rising edge. The second line shifts the current value into the previous-value slot for next scan.

Without rising-edge detection, holding the green PB while the latch is FALSE would re-fire the start every scan as long as the PB stays pressed. With rising-edge, the start only happens once per physical button press. That difference matters when (a) the operator holds the button тАФ you want one start, not 1000 тАФ and (b) the button welds shut from a stray voltage spike тАФ you don't want the motor to auto-restart every time the seal-in tries to drop.

Chunk 2 тАФ Direction-change-while-running detection

DirChangedWhileRunning := RunLatch AND (
    (DirLatch = 1 AND NOT DirFwdSw) OR
    (DirLatch = 2 AND NOT DirRevSw)
);

If the seal-in is held AND the captured direction was FWD but the FWD switch contact is no longer closed, the operator must have moved the selector. Same logic for REV. Either way: the motor is about to be stopped (the drop happens on the next chunk).

Chunk 3 тАФ Drop conditions

DropConditions := (NOT Enable)
               OR (NOT EStopOK)
               OR (NOT DirFwdSw AND NOT DirRevSw)
               OR (DirFwdSw AND DirRevSw)
               OR DirChangedWhileRunning;

Five reasons to drop the latch, OR'd together. Each on its own line so you can read them top-to-bottom: master disable; e-stop unhealthy; selector in OFF position (both contacts open); selector wiring fault (both contacts closed); operator moved the selector mid-run.

Computing DropConditions as a single bool before the IF means the latch logic stays short. If you inlined all five reasons into the IF directly, you'd have a tangle of nested boolean operators that's hard to read and hard to extend. The trade-off cost is one extra named local тАФ well worth it.

Chunk 4 тАФ The seal-in state machine

IF DropConditions THEN
    RunLatch := FALSE;
    DirLatch := 0;
ELSIF StartPBRising AND NOT RunLatch THEN
    IF DirFwdSw THEN
        RunLatch := TRUE;
        DirLatch := 1;
    ELSIF DirRevSw THEN
        RunLatch := TRUE;
        DirLatch := 2;
    END_IF;
END_IF;

Three branches. If any drop condition is true тЖТ kill the latch and clear the captured direction. Else if the button just went high AND we aren't already running тЖТ try to latch in whichever direction the selector points; if neither (selector in OFF), don't latch at all. Otherwise (no drop, no fresh edge): do nothing тАФ the latch holds its existing state.

Order of branches matters Drop comes before start. If both happened in the same scan (e.g., operator presses the button at the exact moment the e-stop is released), the drop wins and the start request is ignored. That's the safer order тАФ you don't want a marginal scan timing to accidentally start a motor.

Chunk 5 тАФ Output assignment

IF NOT RunLatch THEN
    VFDCmdWord := 1;
ELSIF DirLatch = 1 THEN
    VFDCmdWord := 18;
ELSIF DirLatch = 2 THEN
    VFDCmdWord := 20;
ELSE
    VFDCmdWord := 1;
END_IF;
RunLamp := RunLatch;

Latch not held тЖТ STOP. Latch held in FWD тЖТ FWD+RUN. Latch held in REV тЖТ REV+RUN. Defensive fallback to STOP if the latch is somehow TRUE but the captured direction is 0 (shouldn't happen because chunk 4 only sets the latch in a valid direction, but the ELSE is cheap insurance against a future code change). Run lamp mirrors the latch.

9 Scan Cycle Explanation

The Micro 820 runs a 5-phase loop every scan (typically ~10 ms): read inputs тЖТ execute Prog2 тЖТ execute Prog3 тЖТ execute Prog4 (our new one) тЖТ write outputs. Walk one realistic scenario step-by-step:

That's why the keypad takes a beat to display the commanded frequency after you press the button.

10 Testing Procedure

  1. Build (F8). 0 errors. (If errors mention e_stop_ok undeclared, you skipped Step 5's note тАФ add the global + the Prog2 ST line that computes it.)
  2. Download to PLC (USB cable preferred for first download after structure changes).
  3. Switch PLC to Run.
  4. Go online. Variable Monitor тАФ pin vfd_cmd_word, motor_starter_inst.RunLatch, motor_starter_inst.DirLatch.

Test 1 тАФ Green PB while selector in OFF (center) тЖТ no start

Press and release green PB with selector centered. RunLatch stays FALSE. vfd_cmd_word stays 1. _IO_EM_DO_00 stays dark.

Test 2 тАФ Green PB while selector in FWD тЖТ forward start

Flip selector to FWD. Press green PB. RunLatch goes TRUE. DirLatch = 1. vfd_cmd_word = 18. _IO_EM_DO_00 goes solid green. Within ~2 sec the VFD keypad shows the commanded frequency.

Test 3 тАФ Selector to OFF while running тЖТ stop

With motor running forward, flip selector to OFF (center). RunLatch drops to FALSE within one scan. vfd_cmd_word drops to 1. Motor coasts to stop. Run lamp dark.

Test 4 тАФ Direction-change-while-running drops latch (the force-restart behavior)

Restart in FWD (Test 2). While running, flip selector directly from FWD to REV (skipping OFF if your switch design allows). RunLatch drops to FALSE. vfd_cmd_word drops to 1. Motor stops. Run lamp dark. Operator must press green PB again to start in REV.

Test 5 тАФ Verify REV direction actually spins motor reverse

From stopped state with selector in REV, press green PB. vfd_cmd_word = 20. Within ~2 sec motor spins in the reverse direction. If motor spins forward, the deployed REV value (20) is wrong on your firmware тАФ see the bench-verify open question in the carry-forward bug block at the top of this doc.

Test 6 тАФ E-stop drops everything

With motor running, press e-stop. e_stop_active goes TRUE, e_stop_ok goes FALSE, RunLatch drops within one scan. vfd_cmd_word = 1. Motor stops. Contactor (_IO_EM_DO_02) drops open via separate e-stop logic.

Test 7 тАФ E-stop release alone does not restart

Release e-stop. e_stop_active FALSE, e_stop_ok TRUE. RunLatch stays FALSE тАФ motor does NOT auto-restart. Press green PB to restart.

Test 8 тАФ Stuck-PB protection

Simulate stuck PB by force-setting _IO_EM_DI_04 = TRUE in CCW's variable monitor (be careful тАФ this is a real force; release after the test). Motor starts once. Flip selector to OFF (drops latch). Motor stays stopped even though StartPB is still TRUE тАФ the rising-edge prevents auto-restart. Release the force.

12 Mental Model

A UDFB is a sealed starter cabinet. The five labeled terminals on the front of the cabinet тАФ Enable, StartPB, DirFwdSw, DirRevSw, EStopOK тАФ are where you wire the field devices. The two output terminals on the back тАФ VFDCmdWord and RunLamp тАФ are where the cabinet tells the outside world what it has decided to do. Inside is the latching logic, the edge detector, the direction lock. Identical in every cabinet you install. The first time you build one it takes an afternoon; the second cabinet takes ten minutes (drop it on a rung, wire seven pins). The cabinet's internal wiring is the lesson; once you've built one you can build twelve.

13 What You Should Understand Now

  • Why rising-edge is the right trigger for "start". Level-triggered starts allow stuck buttons to auto-restart the motor тАФ unsafe. Rising-edge requires a fresh physical press for each restart.
  • Why drop conditions are computed before the latch IF. Centralizing "everything that should stop the motor" into one named bool makes the body readable and makes adding a new drop condition (say, an over-temperature signal) a one-line change.
  • Why each instance has its own RunLatch. The UDFB's locals are per-instance тАФ that's the leverage of the pattern. Motor A and motor B latch independently with the same code.
  • Why EStopOK is an input, not a hardcoded global reference inside the UDFB. Future flexibility: a second instance could be gated by a different safety chain. Hardcoded globals weld every instance to the same wire.
  • Why VFDCmdWord is an INT output, not a BOOL. The GS10's command word is a bit-field integer (1 / 18 / 20); the UDFB ships the whole integer, not a single bit.
  • Why the direction-change drop is in the drop conditions, not in a separate handler. Anything that should stop the motor goes in DropConditions тАФ one place to read, one place to extend.
  • Why Prog2 ST's vfd_cmd_word := lines had to be commented out. Two writers to the same global within one scan = whichever ran last wins = unpredictable behavior. One writer is the contract for sanity.
  • Why the live monitor shows different values depending on whether you opened the instance or the definition. The definition is the code template тАФ its variable values are meaningless. The instance is the runtime data тАФ its values are what the PLC actually sees this scan.
