1. The bench loop — how to run it through its paces

  1.  CCW: DISCONNECT from the controller, PLC in RUN     (so the Modbus server answers)
  2.  Start a labeled capture for the fault you'll inject:
        python plc/conv_simple_anomaly/live_logger.py --label flaky_photoeye
        (type a note + Enter mid-run to stamp "broke beam now"; Ctrl-C to stop)
  3.  Inject the fault at the bench               (see the "Reproduce" column below)
  4.  Stop the logger → logs/<stamp>_flaky_photoeye.{jsonl,csv} = your ground truth
  5.  Confirm the rule fires:  python plc/conv_simple_anomaly/live_check.py --secs 6
  6.  Ask MIRA the question → it answers grounded in the same live tags
Capture baselines first Record --label baseline_idle (everything stopped) and --label baseline_run_fwd before any fault. Every threshold — motor FLA, DC-bus band, frequency tolerance — is set relative to those golden runs. Without a baseline a number is just a number.
Read-only, always The logger and live_check.py use only Modbus read function codes over Ethernet — side-effect-free per .claude/rules/fieldbus-readonly.md. Nothing here writes to the PLC or the GS10. Inject faults physically at the bench, never by writing registers.

4. What Ask MIRA says

Each answer leads with equipment + fault state, then likely cause, then next check — never a raw-tag wall (per the industrial-HMI rule). Representative answers:

CategoryAsk MIRA reply (grounded shape)

5. Two fixes this catalog surfaces

1 — Reverse cmd-word bug (C1) config.yaml has run_cmd_values: [18, 20] but REV+RUN on the GS10 is 34 (0x20 REV + 0x02 RUN), which you verified at the bench. Until fixed, rules A6 (drive-not-responding) and A10 (freq-frozen) are blind whenever the belt runs in reverse. One-line change to [18, 34] plus a rule test.
2 — Slave-map-v2 reflash Unblocks every REFLASH row at once — the whole photo-eye family (E1–E7) plus A2 GS10-drive-fault decode. The committed map adds DI 5 (coil 000023); the live PLC hasn't been reflashed yet (coil 23 → ILLEGAL DATA ADDRESS). See plc/RESUME_1668_PLC_FEED.md §3.
