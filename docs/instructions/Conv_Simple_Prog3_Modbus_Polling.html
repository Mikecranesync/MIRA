⚠ 3 bugs in plc/LD_BUILD_GUIDE_v5.md (v5.1) that this doc fixes
  1. Channel is wrong: v5.1 says "Channel = 0 (built-in RS-485)" in every rung's config table. The Micro 820's embedded serial port is Channel := 2; channel 0 is reserved for a plug-in serial card that the 2080-LC20-20QBB does not have. GS10_Integration_Guide.md §5 already shows Channel := 2 in the ST code — the build guide drifted from the integration guide and was never reconciled. Fix: set read_local_cfg.Channel := 2, write_cmd_local_cfg.Channel := 2, write_freq_local_cfg.Channel := 2, fault_reset_local_cfg.Channel := 2 in Prog2 ST. Do NOT change the LD rungs themselves — Channel lives in the config struct that Prog2 ST loads each scan.
  2. Rung 1 is reading Output Frequency, not the status word — but the instance is named mb_read_status: v5.1 sets Target Addr = 8451 (0x2103) and reads 6 registers. Per the GS10 register map, 0x2103 is Output Frequency, and the 6-register block 0x2103–0x2108 is the live monitor data (frequency, current, DC bus, output voltage, multi-step, power factor) — useful, but not the operation status word. The actual status bits (running / fault / direction / at-speed) live at 0x2101 (dec 8449). Fix: either rename the instance to mb_read_monitor (recommended — what the v5.1 code is actually reading) and add a second FC03 read at 0x2101 for the status word; or, if you only care about the monitor block, leave it but stop calling it "status".
  3. ElementCnt mismatch — v5.1 contradicts itself, and the safer value is 4: Rung 1's header comment says "reads 4 registers" but the inline config table says ElementCnt = 6. The integration guide ST code (GS10_Integration_Guide.md §5) uses ElementCnt := 4. Per the GS10 register map, the contiguous documented block is 0x2103–0x2106 (Output Frequency, Output Current, DC Bus Voltage, Output Voltage) — exactly 4 registers. Reading 6 would extend into 0x2107 (Multi-Step Speed Sel — valid but rarely meaningful at runtime) and then 0x2108 (undocumented in the GS10 manual register map at §2). Reading undocumented registers is unsupported behavior — the drive may return garbage or a Modbus exception. Fix: use ElementCnt := 4 in Prog2 ST for the monitor read. read_data(1) = Output Freq, read_data(2) = Output Current, read_data(3) = DC Bus, read_data(4) = Output Voltage. If you genuinely need Multi-Step Speed or Power Factor later, add a separate FC03 read for them; don't extend this one.

1 The Big Picture

You're going to make the PLC ask the VFD four questions, once every ~50 ms, forever. "What's your output frequency?" "Run forward at 30 Hz." "Frequency setpoint = 30.0 Hz." "Clear any active fault." Each question is one Modbus packet on the RS-485 wire. Each packet is fired by one ladder rung. A counter (vfd_poll_step) decides whose turn it is to fire — only one rung sends at a time, because you can only have one packet on the wire at a time without collisions.

2 The Real-World Problem

A conveyor sits on the floor with a 2-HP motor wired through a GS10 VFD. The motor's run/stop/direction is controlled from the PLC — not the VFD's keypad. The PLC needs to (a) tell the VFD what to do (run forward at 30 Hz), (b) read back what the VFD is actually doing (motor's at 28.4 Hz, drawing 1.7 A, no fault), and (c) clear faults when the operator presses the reset button. All of this happens over one twisted pair of wires using the Modbus RTU protocol.

The bench rig is one PLC, one drive, one motor, one ~6-foot RS-485 cable. No production gear. You can break things and restart them. That's the point.

3 The Simple Version First

Before the 4-rung sequencer exists, the simplest thing that works is one rung firing one MSG_MODBUS read continuously:

    |                                                            |
    +--[ ]--always_true----[MSG_MODBUS]--mb_read_one-------------+
    |                                                            |

One read, every scan. The PLC fires another Modbus packet before the previous one has finished. The drive ignores you. The serial driver buffers up requests and eventually goes "out of sync". You see CE10 (comm timeout) on the VFD keypad and ErrorID = 55 in the live monitor. The simple version reads one register fine for about 200 ms, then everything jams.

Why the simple version works at all MSG_MODBUS's IN pin is rising-edge triggered. Even with always_true always high, the block only fires once per scan and waits for .Done before re-triggering. So you get one packet every few scans, which "works" for one register. The problem appears the moment you want to read AND write — two messages racing each other.

4 Why This Becomes a Problem

The conveyor needs four conversations: read monitor data, write the run command, write the frequency setpoint, clear faults. If you fire all four from rungs that are all enabled at the same time, the serial driver tries to send four packets stacked end-to-end. The Micro 820's embedded serial driver doesn't queue packets — it expects you to wait for one to complete before starting the next. Stack them and you get partial sends, garbled CRCs, and the drive's CE10 timeout fault.

The pain made concrete Imagine four people trying to ask the same waiter four different questions at the same time. The waiter can answer one at a time. If everyone shouts, the waiter writes nothing down. You need a queue — and a way to take turns.

5 The Better Pattern — a step-driven sequencer

One integer (vfd_poll_step) cycles 1 → 2 → 3 → 4 → 1, advanced by a TON timer in Prog2. Each ladder rung in Prog3 watches the step and fires its MSG_MODBUS only when the step matches its number. So at any given instant exactly one MSG is active. The wire never sees overlap.

A polling sequencer is a rotating spotlight. Only the rung the spotlight is on can speak. When the spotlight moves on, that rung shuts up and the next rung takes its turn. The TON timer is the clock that moves the spotlight; vfd_poll_step is the spotlight's current position.

This pattern is also what lets you add a fifth conversation later (e.g., "read motor temperature") without touching any of the existing rungs — bump the modulo, add a Rung 5, done.

  1. EQU: Source A = vfd_poll_step, Source B = 2.
  2. XIC: vfd_poll_active.
  3. MSG_MODBUS instance: mb_write_cmd. Pins: LocalCfg = write_cmd_local_cfg, TargetCfg = write_cmd_target_cfg, LocalAddr = write_cmd_data.

8 Code Walkthrough — scan by scan

Pick a moment when vfd_poll_step = 2 and vfd_poll_active = TRUE. The state machine just decided we want REV+RUN, so vfd_cmd_word = 20. Walk what happens in one scan:

  1. Input update phase. The Micro 820 reads the physical input table — start button, e-stop, selector. None of those affect Prog3 directly; they affect vfd_cmd_word via the state machine in Prog2.
  2. Prog2 ST executes. Loads all four config structs (Channel=2, Cmd, Addr, Node). Loads write_cmd_data(1) := vfd_cmd_word = 20. Advances poll_timer; if poll_timer.Q, increments vfd_poll_step and resets the timer.
  3. Prog3 LD executes. Rung 1's EQU evaluates vfd_poll_step = 1 → FALSE → rung dead. Rung 2's EQU evaluates vfd_poll_step = 2 → TRUE. XIC sees vfd_poll_active → TRUE. mb_write_cmd.IN rises → the block builds an FC06 packet at 0x2000 with payload 20 and hands it to the serial driver. Rungs 3 and 4: their EQUs are FALSE; their MSGs sit idle.
  4. Output update phase. Nothing in Prog3 drives a physical output, so this is a no-op for our purposes.
  5. Next scan. Packet is still on the wire. mb_write_cmd.Done is FALSE because the drive hasn't replied yet. Rung 2 still satisfies EQU+XIC, but the MSG block won't re-fire because it's still in "waiting for response" state internally.
  6. A few scans later (~5–15 ms depending on baud and packet length), the drive replies. mb_write_cmd.Done goes TRUE for one scan. poll_timer eventually completes; vfd_poll_step increments to 3. Rung 3 takes over.
The reason rung 2 doesn't re-fire while it's waiting for the drive is internal to MSG_MODBUS — the block has its own state machine. EQU+XIC being TRUE is necessary but not sufficient; the block also checks "am I currently in a transaction?" If yes, it ignores its IN until .Done or .Error fires. This is what protects the wire from packet stacking.

9 Scan Cycle Explanation

The Micro 820 runs a fixed five-phase loop every scan (default ~10 ms):

  1. Read inputs. Physical input table copied into the internal image.
  2. Execute Prog2 (ST). Loads configs, advances timer, increments vfd_poll_step.
  3. Execute Prog3-Modbus (LD). Exactly one of the four rungs fires its MSG.
  4. Write outputs. Output image copied to physical outputs.
  5. Housekeeping (comms, diagnostics). The serial driver actually transmits the queued MSG packet during this phase — not during step 3. That's why MSG looks "instant" in the editor but actually takes effect at end-of-scan.

Knowing this matters because: if poll_timer.PT = 50 ms but the scan is 10 ms, vfd_poll_step advances every ~5 scans. Each MSG completes in ~2–3 scans (depends on packet length at 9600 baud). So the rotation comfortably fits within the timer window. If the scan ever stretches to 50+ ms (e.g., heavy logic added elsewhere), the timer fires every scan and MSG completions start colliding with step changes — you'll see ErrorID briefly bouncing through 55 (timeout) until the load drops.

12 Mental Model

The polling sequencer is a rotating spotlight on a stage with four actors. Only the actor in the spotlight can speak. vfd_poll_step is the spotlight's current target. poll_timer is the stagehand who moves the spotlight on a 50 ms cadence. Each MSG_MODBUS instance is one actor, with one line to say into the serial wire. The wire is a stage with room for exactly one speaker — that's why the spotlight exists. Add a fifth actor someday? Just bump the modulo and add a fifth rung. The wire doesn't care; the spotlight handles the choreography.

13 What You Should Understand Now

  • Why the four rungs all look the same and what changes between them (just the step number and the MSG instance).
  • Why Channel := 2 and not 0 for the Micro 820's embedded serial port — and why the deployed Prog2 ST is the authoritative source over the LD build guide when they disagree.
  • That a MSG_MODBUS instance has its own internal "waiting for reply" state — the rung's EQU+XIC enables it, but the block decides when to actually send.
  • Why each rung needs its own MSG instance and its own pair of cfg structs (sharing creates wire collisions and mid-scan struct mutation races).
  • The difference between the status WORD (register 0x2101) and the live monitor BLOCK (0x2103+) — v5.1 conflated them and called the monitor read "status".
  • Why the frequency value is multiplied by 10 (the GS10's frequency register is encoded as Hz × 10, integer).
  • How the scan loop's five phases interact with MSG: the rung enables the block during phase 3, but the actual serial transmit happens in phase 5.
  • What the timer cadence vs scan time relationship is and why a slow scan eventually corrupts the polling sequence.

15 Open question — bench-verify before relying on production

Command word bit math has a known contradiction between Mike's sources Three different values for "REV + RUN" appear across the existing documentation:
  • 20 (0x0014) — plc/LD_BUILD_GUIDE_v5.md Rung 2; GS10_Integration_Guide.md §2 "Common Command Words" table; deployed Prog2 ST.
  • 18 (0x0012) — same sources, but labeled as FWD+RUN there.
  • 34 (0x0022) — the existing ~/.claude/skills/plc-instruction-guide/references/style-vocabulary.md bug example says this is the correct REV+RUN value with explicit bit math "bit 5 + bit 1 = 32 + 2 = 34".
Per GS10_Integration_Guide.md §2's bit-field table: RUN sets bit 1 (decimal 2), FWD sets bit 3 (decimal 8), REV sets bit 4 (decimal 16). That arithmetic gives FWD+RUN = 10 and REV+RUN = 18. Neither matches the "Common Command Words" table on the same page, which lists FWD+RUN = 18 and REV+RUN = 20. The skill's bug example asserting 34 corresponds to a different bit-position convention again (bit 5 for REV).

What to do: bench-test all three values when you sit down at the rig. Whichever value actually spins the motor in reverse on your specific GS10 firmware is the correct one. Update this guide's cheatsheet and the integration guide with the bench-verified value and a [verified on bench YYYY-MM-DD] citation. Then delete the wrong values from the cheatsheet and from the v5.1 build guide so they don't propagate further. This is exactly the "when sources disagree" workflow in ~/.claude/skills/plc-instruction-guide/references/citation-discipline.md.
