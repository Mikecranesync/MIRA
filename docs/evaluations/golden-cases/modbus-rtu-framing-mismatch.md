# Golden Case: Modbus RTU Framing Mismatch (Micro820 ↔ GS10)

**Case ID:** GC-2026-05-24-modbus-rtu-framing

**Equipment:** Allen-Bradley Micro820 (2080-LC20-20QBB), AutomationDirect GS10 VFD
**Category:** modbus-troubleshooting / serial-comms
**Difficulty:** hard (multi-source cross-reference required)
**Status:** Reference case — MIRA must reproduce this diagnosis from KB + photos + sniffer data.

## 1. Problem statement

> *"Modbus RTU communication works when the USB-to-RS485 adapter is the master querying the GS10 (or querying the PLC over Ethernet TCP). It fails PLC-to-VFD direct on the same physical wiring."*

Physical layer is good. Logical layer is broken on one specific link.

## 2. What the technician asked (paraphrased — full session above)

1. **"Why does it work with the USB adapter but not without?"**
   → Sniffer with adapter-as-master uses adapter's framing; direct PLC↔GS10 uses each endpoint's *configured* framing. The two configs must match.
2. **"What are the correct pin assignments for the GS10 RJ45?"**
   → GS10 manual Ch. 5, pins 3/4/5 = SGND / SG- / SG+.
3. **"I'm reading +5V between blue/white and blue — what does that mean?"**
   → Idle differential voltage on the SG+/SG- pair. Healthy bias from one transceiver. Cable + bias resistors are fine.
4. **"The sniffer shows 0x3F on every byte with CRC Invalid — what's wrong?"**
   → 0x3F is the canonical UART symptom of framing mismatch (parity or stop-bit count wrong). NOT bus noise — noise produces random bytes, not a repeating value.
5. **"P09.04 was set to 13, is that right for my setup?"**
   → No. P09.04 = 13 selects 8-N-2 framing. CCW Serial Port had the Micro820 set to 8-E-1. Change P09.04 to 14 (8-E-1).

## 4. What MIRA should deduce

Given (a) CCW screenshot showing 38400 / 8-E-1 / Stop=1 on the Micro820 serial port and (b) GS10 P09.04 = 13:

1. P09.04 = 13 → 8-N-2 (No parity, 2 stop bits) per GS10 Ch. 4 table.
2. PLC is sending 8-E-1; VFD expects 8-N-2.
3. Both frames are 11 bits total, so the UART does not raise a framing error — it shifts the data bytes by one bit and CRC fails.
4. Symptom: every byte decodes as 0x3F (or some other repeating-pattern byte), CRC always invalid.
5. Root cause: **framing mismatch**, not noise, not address, not turnaround timing, not register selection.
6. Fix: change P09.04 from **13 → 14** (8-E-1 RTU) on the GS10 keypad. No PLC change required.

## 8. Adversarial / failure modes to test

MIRA should NOT recommend any of these (they are common ungrounded-LLM responses):

- "Add a 120 Ω termination resistor" (irrelevant — symptom is framing, not reflection)
- "Reduce baud to 9600" (works by accident sometimes — masks the real bug)
- "Check the cable polarity / swap A and B" (already verified by +5V reading)
- "It's electrical noise — add ferrites" (wrong — 0x3F is a deterministic pattern)
- "Reset the drive to factory defaults" (would change P09.04 to 13 — *causes* the bug)
- "Change the Micro820 to 8-N-2 in CCW" (works mathematically but loses Even-parity error detection; the *correct* direction is to make the drive match the controller, not vice versa)
