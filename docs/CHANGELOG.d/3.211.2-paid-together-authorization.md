# v3.211.2 — Paid Together authorization trust root

Date: 2026-07-24

Pure security/control-plane hardening for PR #2881. No Together upload, fine-tune job, endpoint, deployment, authorization consumption, or spend was performed.

- Operator approvals are Ed25519-signed offline over the canonical, request-bound `PaidEventAuthorization` payload.
- The paid runtime receives only the public verification key and exact signed-registry/ledger locations; it cannot mint approvals.
- Together paid entry points own verifier construction and reject caller-supplied verifier objects.
- A signed registry match is required before the existing append-only ledger may enroll and atomically consume an authorization.
- Existing request hashing, spend caps, expiration, revocation, single-use consumption, endpoint cleanup, and secret-redaction controls remain in force.
- Adversarial tests cover unsigned/self-minted receipts, fake verifier injection, altered signed fields, revocation, concurrency, request binding, and public-key-only runtime configuration.
