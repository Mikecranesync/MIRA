## Executive summary

MIRA's chat critical path works (47/57 eval baseline, eight production containers green) but the foundation it sits on has drifted. Three of the largest hot-path files have bloated past any reasonable single-responsibility line: `engine.py` is **2,531 lines**, `mira-core/mira-ingest/main.py` is **1,621**, and `mira-pipeline/main.py` is **1,331** — that last one was ~500 in the April 14 elegance audit, a 2.6× expansion in 11 days. Configuration is scattered: timeouts appear as literals in 15+ places, hourly provider budgets are baked into a class dict, and provider order is hardcoded with comments explaining historical reordering. The chat path passes through three different cascade implementations (router, RAG worker, engine) that don't share failover logic.

The April 14 elegance audit flagged most of these problems. Some Tier 1/2 cleanups have been executed (mira-hud and mira-prototype archived) but several have not (`paperclip/`, `mira_copy/`, `bravo/`, `ignition/` still on disk; Teams/WhatsApp/Reddit adapters still build). Worse, since April 14 a new module appeared — **mira-hub** (Next.js CMMS rebuild, 14 in-flight feature branches per `pre-merge-review-2026-04-25.md`) — and it is **not in the CLAUDE.md repo map at all** (`grep -c mira-hub CLAUDE.md` returns 0). Onboarding agents have no signpost to it.

The good news: most of the highest-priority items are small. Eleven of the twelve sprint-window fixes are S-effort (≤1 day each). They unblock the larger architectural moves that have to land inside the 90-day MVP window so the parity-sprint branches can merge cleanly.

**Top three risks if nothing changes:**

1. **Cascade duplication × magic timeouts** — A provider degradation incident (e.g. Groq the next time it 503s) requires hunting changes through router, RAG, and at least 15 timeout literals. Last time Gemini moved to position 3 after persistent 503s; the comment is in router.py because there was no other knob.
2. **mira-hub merging into a documentation void** — 14 feature branches with 18 reviewer-flagged issues are about to land on a `main` whose canonical repo map doesn't acknowledge the module exists.
3. **Engine.py and ingest main.py keep absorbing scope** — Both grew significantly since the April 14 audit. A god-object that's already 2,531 lines and gaining will make every parity-sprint feature touch the same file, multiplying merge conflict surface area.

## Methodology

Three parallel read-only scans (pipeline+bots, mcp+ingest, supporting modules) cross-referenced with `docs/known-issues.md`, `docs/QUALITY_SCORE.md`, the April 14 elegance audit, and the April 25 pre-merge review. Then top findings were spot-verified against the current code (file sizes, line numbers, presence-on-disk) before scoring.

**Priority formula** (from `engineering:tech-debt` skill):

> Priority = (Impact + Risk) × (6 − Effort)

with Impact and Risk on 1–5, and Effort mapped S=1, M=2, L=3, XL=5.

Foundation-first sequencing (per project memory) was applied: items that other items depend on are placed earlier even when raw priority score is similar.

## Cross-cutting observations

**The April 14 audit is mostly still true.** Two of its top recommendations have been executed (mira-hud, mira-prototype archived). Most have not. Re-running the audit on April 25 would produce a similar list, plus mira-hub. The pattern matters: every audit that finds the same thing is an audit that didn't ship anything. Phase-1 of this report is short on purpose so it can actually land before the next audit.

**Phase 1 unblocks Phase 2 unblocks Phase 3.** The Supervisor split (DBT-001) is impossible while engine.py imports the same monolithic guardrails.py and is imported by a pipeline that uses a sys.path hack. Fixing the sys.path hack (DBT-003) and splitting guardrails (DBT-011) are prerequisites; doing the Supervisor refactor first means redoing it after every other change. Hence the foundation-first sequencing: archive dead code → centralize config → harden boundaries → then refactor the god objects.

**The 18 reviewer bugs in pre-merge-review-2026-04-25.md are the same shape as several items here.** Seven of them (KEK rotation, audit-log immutability, libsodium dedup, PII sanitizer, schema-drift risk, npm-deps gate, auth-stub sweep) map directly to DBT items in this register. Treat the parity-sprint reviewer fixes and these tech-debt items as one program of work — split-tracking them invites duplicate effort.

**No security-debt items in this list by design.** The user scoped this audit to Code + Architecture only. The pre-merge review covers security comprehensively; do not let this report's silence on auth/RLS/secrets be read as "those problems don't exist." They do, and they're tracked separately.
