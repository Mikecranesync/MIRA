## Summary

Ships the QR asset-tagging MVP end-to-end: a technician scans a printed QR sticker on a piece of equipment → lands on an authenticated scan route in mira-web → pipeline picks up the pending-scan cookie on next chat turn → `Supervisor.process()` sees the asset context from the scan and doesn't ask "what machine are you working on?"

Resolves spec `docs/superpowers/specs/2026-04-19-mira-qr-system-design.md`. Follows plan `docs/superpowers/plans/2026-04-19-mira-qr-system-plan.md` (tasks 1–8; task 9 is physical dogfood).

**Also brings in PR #401 (#329 cross-session memory)** via merge from origin/main — was prerequisite infrastructure for Task 5 (the pipeline bridge calls `save_session`). The 4 session_memory files are part of this diff because the branch pulled main.

## Architecture

```
Phone scans QR  →  GET /m/:asset_tag (mira-web, requireAuth)
                      ├─ Validate tag (regex)
                      ├─ Constant-time tenant-scoped resolution
                      ├─ Log scan_event (audit) — every scan, found or not
                      ├─ If found: Set-Cookie mira_pending_scan=<scan_id> (5-min) → 302 /c/new
                      └─ If not found: byte-identical 200 HTML (no tenant-enumeration oracle)

Open WebUI chat →  POST /v1/chat/completions (mira-pipeline)
                      ├─ qr_bridge.process_pending_scan(cookie, chat_id)
                      │    ├─ Read + validate pending_scan UUID
                      │    ├─ Lookup scan row in qr_scan_events
                      │    └─ session_memory.save_session(chat_id, asset_tag)
                      ├─ engine.process(...) — existing IDLE load_session hook
                      │    picks up the asset context from #329
                      └─ If seeded: Set-Cookie mira_pending_scan=; Max-Age=0 (one-shot)

Admin UI      →  GET /admin/qr-print   (requireAdmin) — pick assets → PDF
                GET /admin/qr-analytics (requireAdmin) — scan_count, last_scan
                POST /api/admin/qr-print-batch — UPSERT asset_qr_tags + generate Avery 5163 PDF
```

## Security properties (spec §12.6)

- **Constant-time tenant resolution.** Same SELECT regardless of whether the scan hits a row for this tenant, another tenant, or doesn't exist. Branch only on the result, never before.
- **Byte-identical not-found HTML.** Cross-tenant miss and nonexistent tag return the exact same bytes. No enumeration oracle.
- **Input validation at the edge.** `ASSET_TAG_RE = /^[A-Za-z0-9._-]{1,64}$/` gates the route before any DB call. Path traversal (`../../etc/passwd`) returns 400.
- **Cookie hygiene.** HttpOnly + SameSite=Lax + Secure (in non-dev) + Domain=.factorylm.com. `mira_pending_scan` is 5-min TTL and cleared on first successful chat-turn. `mira_session` is 30-day.
- **Audit trail.** Every scan (found OR not-found) logged to `qr_scan_events` with tenant_id, asset_tag, atlas_user_id, user_agent, scan_id, timestamp. 90-day retention via `purge_qr_scan_events.py`.
- **UUID validation on pipeline side.** `qr_bridge.read_pending_scan_id` rejects non-UUID cookie values before hitting NeonDB. Junk-in-cookie can't reach the DB.
- **Admin role gated.** `requireAdmin` checks `atlasRole === 'ADMIN'` — old tokens (pre-claim) fall through to 403. Users must re-login to get admin access.
