## Runtime triggers are NOT wired (intentional, documented)

P5 (`TagDiffLogger`) and P7 (`FlakyInputDetector`) ship **logic + store boundary +
tests**. Neither is yet invoked by a cron/worker on the live stream. "Done" here
means the components exist and are verified — NOT that they are running in
production. Wiring them to a schedule (a relay worker, or a Celery beat task that
calls `TagDiffLogger.process_batch` / `FlakyInputDetector.run` against the live
window) is the natural next step and is additive — no interface change needed.
`NeonDiffStore` / `NeonFlakyStore` are the prod stores to inject.

## Decision-trace evidence: what's captured vs schema-ready

`decision_traces` columns and how they're populated today:

- **Captured every turn:** `uns_path` / `source` / `confidence` (from post-gate
  `state["context"]["uns_context"]`), `user_question` + `recommendation`
  (PII-sanitized), `citations_present` (from `[Source:]` tags), `manual_evidence`
  (RAG `_last_sources`, **only when the turn actually retrieved** — guarded
  against stale carry-over), `platform`, `latency_ms`, `outcome` (resolved /
  gate_fired / null).
- **Captured for Ignition turns:** `tag_evidence` (from `req.tag_snapshot`).
- **Schema-ready but NOT populated by current wiring:** `tag_evidence` for
  non-Ignition surfaces, and `kg_evidence` (the KG traversal results aren't at
  `engine.process()` scope). These columns default to `[]` — do not read an
  empty column as "captured nothing"; read it as "this surface doesn't feed it
  yet." Wiring `kg_evidence` means surfacing the engine's KG-traversal hits up to
  `process()`; a follow-up.
- `session_id`, `technician_confirmed`, `model_used` are nullable and currently
  null at this layer (filled later / by other surfaces).

## Hub `/proposals` wiring (P8/P12)

`mira-hub/src/app/api/proposals/route.ts` reads `relationship_proposals` LEFT
JOINed to `kg_entities` for source/target names, and counts `relationship_evidence`.
So a flaky-detector proposal **renders automatically** with `evidence_count > 0`
— PROVIDED both `source_entity_id` (signal) and `target_entity_id` (asset/component)
are real `kg_entities` rows. The detector enforces this: `build_proposal_spec`
returns `None` (alert-only, no dangling proposal) unless the rule supplies both
ids. For the bench, `flaky_rules.json` leaves those ids `null` (entities are
seeded per-install) → the detector writes the **alert + `ai_suggestions` row**
but opens the `relationship_proposals` edge only once the rule is populated with
real entity ids. **Action for the operator:** to see the full KG loop in Hub
`/proposals`, seed `kg_entities` for the bench signal + asset and fill
`signal_entity_id` / `equipment_entity_id` in `flaky_rules.json`.

Note: the route reads `relationship_proposals`, NOT `ai_suggestions`, so the
`ai_suggestions` kg_edge header the detector also writes is currently a
parallel record (the broader "Hub /proposals should also render `ai_suggestions`
non-edge types" gap is a separate Solve-stage item, out of this scope).

## Risks / what needs a real environment before merge

1. **Migration 037 + all three Neon store SQL paths have only run against
   in-memory test doubles.** `tag_diff_logger.NeonDiffStore`,
   `flaky_detector.NeonFlakyStore`, and `decision_trace._insert` SQL have never
   touched a live/ephemeral Postgres. The column names line up with the verified
   032/033/034/018/027 schemas, but NULL-LTREE/UUID casts and the multi-table

   unexecuted path bites. **Verify via `migration-verify.yml` on push** (037
   applies) and apply 037 to the ephemeral pg the way 032–036 were verified.
2. **`direct_connection` marker vs gate — now honored, but only at the chat
   gate.** `f0106f15` makes `_should_fire_uns_gate` return False on
   `source="direct_connection"`, so the marker no longer contradicts
   `direct-connection-uns-certified.md`. The broader contract pieces still
   deferred to master-plan **P6**: rejecting a direct-connection turn that
   arrives WITHOUT a resolvable identifier (`{"error":"uns_required"}`), and the
   `confidence="certified"` band. Today an asset-less Ignition turn is treated as
   a plain chat turn, not rejected.
3. **Engine `tag_evidence`/`kg_evidence` non-Ignition population** — see above.
4. **Migration numbering** — `037` assumes `origin/main` hasn't grown its own
   030+. If it has, rename per the gap-plan `b`-suffix collision rule at merge.
